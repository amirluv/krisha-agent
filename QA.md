# KrishaAgent — Q&A для защиты (Demo Days)

Ответы на контрольные вопросы из раздела 6.2 финального ТЗ.

---

## Архитектура и техническое решение (40%)

**— Почему именно LangGraph? Что рассматривали как альтернативу?**

Рассматривал три варианта:
- **LangChain LCEL** — нет встроенной поддержки циклов и human-in-the-loop. После `generate_recommendation` нужно ждать вопроса пользователя и отвечать в том же контексте — LCEL это не умеет нативно.
- **CrewAI** — заточен под мультиагентные системы с ролями. У меня один агент, и CrewAI добавил бы лишнюю абстракцию и сложность в отладке.
- **LangGraph** — явный граф состояний: каждая нода изолирована, тестируема, видна в LangSmith как отдельный span. Ветвление после `fetch_listing` и human-in-the-loop реализуются в 5 строк.

---

**— Как устроен поток управления? Где ветвления, циклы, human-in-the-loop?**

```
fetch_listing
  ├── ошибка (404, блокировка) → END  ← ветвление
  └── ок → search_similar → analyze_photos → evaluate_price
              → generate_recommendation → human_review → END
```

**Ветвление:** `_listing_loaded()` — conditional edge после первой ноды. Если `state["error"]` не None — сразу END, пользователь видит сообщение об ошибке.

**Human-in-the-loop:** нода `human_review` ждёт `user_feedback`. Пользователь задаёт вопрос через Streamlit → вызывается `answer_followup(state, question)` → граф переходит в `human_review` с новым фидбэком → GPT-4o-mini отвечает в контексте → рекомендация дополняется.

---

**— Проведите по пути одного запроса от пользователя до ответа.**

1. Пользователь вставляет `https://krisha.kz/a/show/1010382901`
2. `fetch_listing` — GET запрос, BeautifulSoup парсит заголовок, цену, площадь (65м²), этаж (7/14), район (Бостандыкский), 10 URL фото
3. `search_similar` — формирует запрос `"2-комнатная квартира Бостандыкский 65м²"`, ChromaDB возвращает 5 похожих из 36 686
4. `analyze_photos` — скачивает все 10 фото, отправляет GPT-4o Vision одним запросом → `condition_score=7`, `renovation_needed="не нужен"`, `price_adjustment_pct=+3%`
5. `evaluate_price` — SQLite: выборка 2-комн Бостандыкский ±30% площади → IQR фильтрация → `avg_m2=750 000₸`. Корректировки: монолит 2017 +2%, средний этаж 0%, ремонт +3%. fair_m2=772 500₸, диапазон 47.5–55.8млн. Объявление 50млн → **рыночная цена**
6. `generate_recommendation` — GPT-4o собирает всё в структурированный текст с цифрами
7. `human_review` — ожидает follow-up вопрос

Общее время: ~20 секунд.

---

**— Какие компоненты независимы и могут быть заменены?**

**Легко заменяемые (слабая связность):**
- SQLite → PostgreSQL / любая SQL БД (интерфейс через `database.py`)
- ChromaDB → Qdrant / Pinecone (интерфейс через `retriever.py`)
- GPT-4o → Claude 3.5 / Gemini (один параметр в `config.py`)
- Streamlit → любой фронтенд (агент — отдельный модуль `agent/`)

**Сознательно выбрал coupling:**
- Ценовой алгоритм в `evaluate_price` жёстко привязан к структуре данных SQLite — сделано специально, чтобы каждый коэффициент (год -35%, панель -10%) был задокументирован в коде, а не скрыт внутри промпта модели.

---

**— Почему GPT-4o? Как сбалансированы стоимость, скорость, качество?**

Использую **две модели** с разным назначением:

| Задача | Модель | Почему |
|--------|--------|--------|
| Vision анализ фото | gpt-4o | Единственный реальный выбор — нужно мультимодальное качество |
| Рекомендация | gpt-4o | A/B тест показал — mini даёт сопоставимое качество |
| Follow-up вопросы | gpt-4o-mini | Простой диалог в контексте, mini справляется |
| Эмбеддинги | text-embedding-3-small | В 5× дешевле large, качества достаточно |

Почему не Claude: требует отдельного SDK, нет такого же качества Vision API для русскоязычного контента. Почему не Gemini: тестировал — хуже понимает русские описания недвижимости.

---

**— Сколько стоит один запрос? Что делали для оптимизации?**

Стоимость одного запроса: **~$0.10–0.15**
- Vision (10 фото): ~$0.08–0.12
- Рекомендация (800 токенов): ~$0.01–0.02
- Эмбеддинги: ~$0.0001

**Оптимизации:**
- Top-5 похожих в контекст рекомендации, не все найденные
- Vision делает один запрос на все 10 фото, не 10 отдельных — экономия на overhead
- Follow-up вопросы через gpt-4o-mini (в 15× дешевле gpt-4o)
- Вердикт вычисляется алгоритмически, не моделью — нет лишнего LLM-вызова

---

**— Что если основная LLM недоступна или резко подорожает?**

Сейчас явного fallback нет — честный ответ. При недоступности OpenAI агент упадёт с ошибкой. Стратегия для production:
- Vision fallback: если GPT-4o Vision недоступен — пропустить `analyze_photos`, выставить `photo_adjustment_pct=0`
- Recommendation fallback: переключиться на gpt-4o-mini — A/B тест уже показал сопоставимое качество при цене в 10× меньше

---

**— Какие trade-off осознанно приняли?**

1. **Качество vs Latency:** Vision анализ всех 10 фото — ~8 секунд. Сознательный выбор: без фото оценка ремонта неточная.
2. **Точность vs Простота:** детерминированный ценовой алгоритм вместо "пусть GPT решит". Теряем гибкость, получаем предсказуемость и объяснимость каждой цифры.
3. **Локальная БД vs Real-time:** SQLite с 37k объявлений вместо live скрапинга. Теряем актуальность, получаем скорость и надёжность.
4. **Streamlit vs React:** нет красивого UI, но весь бюджет ушёл в качество агента.

---

**— Какие гипотезы не подтвердились?**

1. **"Live Krisha API будет работать"** — Krisha блокирует автоматические запросы. Итог: похожие объявления только из RAG.
2. **"GPT-4o даст значительно лучшие рекомендации чем mini"** — A/B тест опроверг: accuracy одинакова (86.7%), relevance у mini даже выше. Урок: для структурированного вывода с готовыми данными mini достаточно.
3. **"Освещённость важна для оценки квартиры"** — модель путала тёмные фото с плохим ремонтом. Убрал освещённость из метрик, добавил флаг `low_confidence`.

## Качество LLM-инженерии (25%)

**— Как эволюционировал главный промпт Vision? v1 vs текущая версия.**

**v1:**
```
"Оцени квартиру по фото. Определи:
1. Состояние ремонта (1–10)
2. Планировка (хорошая / средняя / плохая)
3. Освещённость
4. Видимые проблемы
5. Влияние на рыночную стоимость (%)
Ответ строго в JSON."
```
Проблема: модель ставила 7–8 почти всем квартирам. Освещённость влияла на оценку — тёмное фото = плохая оценка.

**Текущая версия — что изменилось:**
- Жёсткие якоря шкалы: `"3-4: советский фонд, подъезд убитый, обои отваливаются"`
- Явная инструкция: `"Большинство квартир получают 4-6, только реально хорошие 7+"`
- Убрана освещённость
- Добавлено: `"НЕ путай качество съёмки с состоянием — тёмное фото ≠ плохой ремонт"`
- Добавлен флаг `low_confidence`: если фото нечёткие — не штрафуем
- Корректировка цены вынесена в детерминированную функцию `_score_to_price_adj()` — убрали галлюцинации в числах

---

**— Что в golden dataset? Как покрываются edge cases?**

30 примеров, 3 класса:
- 11 × выгодно: новостройки 2019–2024, цена ниже рынка, хороший ремонт
- 13 × дорого: хрущёвки 1950–1975, аварийное состояние, первый/последний этаж при завышенной цене
- 6 × рыночная цена: граничные случаи, разные районы

**Специально включённые edge cases:**
- Дом 1940 года — проверяем максимальный дисконт -35%
- Первый этаж + хороший ремонт — корректировки в разные стороны
- 1955 год в Медеуском — очень низкая цена для дорогого района → "выгодно"
- Панельный дом с евроремонтом — -10% за панель vs +10% за ремонт

---

**— Какие метрики измеряете? Что они НЕ показывают?**

**Verdict Accuracy (86.7%)** — правильность вердикта. НЕ показывает качество объяснения.

**Relevance Score (0.79–0.85)** — покрытие ключевых факторов. НЕ показывает читабельность, работает только через keyword matching.

**LLM Judge (4.90–4.97/5)** — качество рекомендации глазами другой модели. НЕ показывает реальную полезность для пользователя. Может быть предвзятым в пользу длинных ответов.

---

**— Расскажите про A/B эксперимент.**

**Гипотеза:** GPT-4o для финальной рекомендации даст значительно лучший LLM-judge score чем gpt-4o-mini, и это оправдает 10× рост стоимости.

**Что сравнивали:** Config A (gpt-4o-mini) vs Config B (gpt-4o), 30 примеров на каждый.

**Результат:**
- Accuracy: 86.7% vs 86.7% — одинаково (вердикт детерминирован, модель не влияет)
- Relevance: 0.851 vs 0.794 — mini лучше
- LLM Judge: 4.97 vs 4.90 — mini лучше

**Решение:** гипотеза не подтвердилась. GPT-4o остаётся только для Vision. Для рекомендации переходим на mini — снизит стоимость запроса в ~5×.

---

**— Как устроен RAG? Какие альтернативы пробовали?**

- **Chunking:** 1 объявление = 1 чанк. Добавлял описание — не дало прироста, только увеличило индекс.
- **Embeddings:** `text-embedding-3-small`. Пробовал `ada-002` — хуже на русских коротких текстах.
- **Vector DB:** ChromaDB локально. Пробовал Qdrant — требует отдельный сервер, избыточно для 36k документов.
- **Retrieval:** top-5 по косинусному сходству. Reranker не добавлял — для 5 результатов overhead не оправдан.
- **Баг дедупликации:** у RAG-объявлений `url=""`, все 5 результатов склеивались в 1. Исправил: ключ дедупликации — `title` если url пуст.

---

**— Какие edge cases ломают систему сейчас?**

1. **5-комнатные квартиры** — мало данных в базе, агент сравнивает со всем районом → некорректная статистика.
2. **Объявления без фото** — `analyze_photos` возвращает None, `photo_adjustment=0%`. Не ломается, но точность ниже.
3. **Районы вне 8 поддерживаемых** — fallback на статистику по всему Алматы.
4. **Цены в USD** — парсер ожидает тенге, USD объявления дают некорректную цену м².

## Полнота покрытия обязательных модулей (20%)

**— Покажите MCP-сервер. Почему эти tool'ы и почему MCP, а не обычный API?**

Три tool'а закрывают три разных вопроса агента: *что продаётся?* (search_listings), *какой рынок?* (get_price_statistics), *справедлива ли цена?* (estimate_fair_value). Каждый атомарен и используется независимо.

MCP vs обычный API: MCP автоматически генерирует JSON-схему инструментов, которую LLM понимает без дополнительного кода. Можно подключить к Claude Desktop или Cursor — пользователь спрашивает "найди 2-комнатные в Бостандыке до 50 млн" и получает ответ через наш сервер без изменений кода.

---

**— Покажите Skill. Когда Skill полезнее обычного промпта?**

SKILL.md содержит: описание агента, триггеры (ключевые слова и форматы URL), полную схему входных/выходных данных, описание всех 6 нод, таблицу моделей с обоснованием.

Skill полезнее промпта когда агент нужно запустить из внешней системы — оркестратор читает SKILL.md и знает: при ключевых словах "krisha", "купить квартиру" нужно вызвать этот агент, передать URL, ожидать структурированный AgentState. Обычный промпт этого не описывает.

---

**— Где используется мультимодальность? Что потеряли бы без неё?**

Vision в ноде `analyze_photos`: все 10 фото (интерьер + подъезд) → GPT-4o → `condition_score` 1–10.

Без Vision: агент знал бы только цену, площадь, этаж, год. Продавцы пишут "хороший ремонт" даже для хрущёвки с советской сантехникой. Фото не лгут — убитый подъезд и трещины на стенах реально снижают стоимость.

Конкретный пример: квартира 1937 года, описание "ремонт есть" → Vision видит разрушенную штукатурку → `condition_score=3` → `-8%` к цене → вердикт "дорого" вместо "выгодно".

## Бизнес-ценность (5%)

**— Кто конкретный пользователь?**

Человек 25–40 лет, смотрит объявления на Krisha.kz для покупки первой квартиры в Алматы, не имеет экспертизы в оценке недвижимости и боится переплатить.

**— Какую боль закрываем? Как решают сейчас без продукта?**

Без агента: открывают 20–30 объявлений, вручную сравнивают цены м², звонят риелтору (который заинтересован в продаже), просят знакомых. Часы работы, субъективный результат.

С агентом: вставил ссылку → 20 секунд → вердикт с цифрами, анализ фото, рекомендация по торгу.

**— Пробовали давать реальным людям?**

Тестировал на реальных ссылках в процессе разработки. Основной фидбэк: для типичных 1–3 комнатных оценка совпадает с интуицией. Edge case — 5-комнатные иногда оцениваются некорректно из-за малой выборки.

**— Что должно произойти чтобы продолжить после курса?**

Если 10 человек использовали агент при реальной покупке и это повлияло на решение — достаточный сигнал. Следующий шаг: автообновление базы раз в неделю + Telegram бот вместо Streamlit.
