# KrishaAgent — AI-агент для анализа квартир на Krisha.kz

**Проблема:** Покупатель квартиры в Алматы не знает — переплачивает он или нет. Ручной анализ десятков объявлений занимает часы, а оценить справедливость цены без эксперта практически невозможно.

**Решение:** Вставляешь ссылку объявления с Krisha.kz — агент скрапит данные, сравнивает с базой 37 763 реальных объявлений, анализирует все фото квартиры через GPT-4o Vision и выдаёт вердикт: **выгодно / рыночная цена / дорого** с конкретным обоснованием и рекомендацией по торгу.

## 1. Архитектура агента (LangGraph)

**Почему LangGraph, а не простая цепочка вызовов:**
Задача требует условной логики (что делать если объявление не загрузилось?), асинхронного сбора данных из разных источников и human-in-the-loop после выдачи рекомендации. LangGraph позволяет описать это как явный граф состояний с ветвлениями — каждый шаг изолирован, тестируем и виден в LangSmith как отдельный span.

**Альтернативы, которые рассматривались:**
- `LangChain LCEL` — нет встроенной поддержки циклов и human-in-the-loop
- `CrewAI` — избыточен для одного агента, сложнее трейсить
- Простые функции — нет наблюдаемости, нельзя прервать/возобновить

### Граф (6 нод):

```
START
  │
  ▼
[fetch_listing]         ← скрапит krisha.kz/a/show/<id>
  │
  ├── ошибка загрузки ──→ END
  │
  ▼
[search_similar]        ← RAG + Krisha API, top-5 похожих
  │
  ▼
[analyze_photos]        ← GPT-4o Vision, все фото
  │
  ▼
[evaluate_price]        ← SQLite статистика + корректировки
  │
  ▼
[generate_recommendation] ← GPT-4o, итоговый текст
  │
  ▼
[human_review]          ← ожидает вопрос пользователя
  │
  ▼
END
```

**Ветвление:** после `fetch_listing` — если объявление не удалось загрузить (404, блокировка, неверный URL), граф уходит в `END` с сообщением об ошибке. Пользователь не видит пустой экран.

**Human-in-the-loop:** нода `human_review` ожидает follow-up вопрос. Пользователь может спросить *"стоит ли торговаться?"* или *"что скрывает продавец?"* — GPT-4o-mini отвечает в контексте уже выполненного анализа без повторного запуска всего пайплайна.

**State** передаётся между нодами как TypedDict:

```python
class AgentState(TypedDict):
    listing_url: str
    listing_data: dict        # данные объявления
    similar_listings: list    # похожие из RAG
    photo_analysis: dict      # результат Vision
    price_stats: dict         # статистика рынка
    price_evaluation: dict    # вердикт + корректировки
    market_comparison: dict   # место среди похожих
    recommendation: str       # текст рекомендации
    user_feedback: str        # вопрос пользователя
    iteration: int            # защита от зацикливания
    error: str | None
```

**Файлы:** [agent/graph.py](agent/graph.py), [agent/nodes.py](agent/nodes.py), [agent/state.py](agent/state.py)

## 2. MCP-сервер

**Что такое MCP и зачем он здесь:**
MCP (Model Context Protocol) — стандарт для описания инструментов, которые LLM может вызывать. В отличие от обычного API, MCP автоматически генерирует схему инструментов, которую модель понимает. Это позволяет подключить сервер к любому MCP-совместимому клиенту (Claude Desktop, Cursor и др.) без изменения кода.

**Запуск сервера:**
```bash
fastmcp run mcp_server/krisha_mcp.py
```

### Tool 1: `search_listings`
Ищет объявления по параметрам через Krisha API.

```python
search_listings(
    district="Бостандыкский",  # район Алматы
    rooms=2,                    # количество комнат
    price_min=30000000,         # мин. цена в тенге
    price_max=60000000,         # макс. цена в тенге
    limit=10                    # кол-во результатов
)
# → {"listings": [{title, price, area, price_m2, url}, ...]}
```

### Tool 2: `get_price_statistics`
Возвращает статистику рынка: среднюю, медианную цену м², тренд за 4 недели.

```python
get_price_statistics(
    district="Медеуский",
    rooms=3
)
# → {avg_price_m2: 920000, median: 900000, trend_4w_pct: +0.3, ...}
```

### Tool 3: `estimate_fair_value`
Рассчитывает справедливую стоимость квартиры с учётом площади, этажа и состояния.

```python
estimate_fair_value(
    area_sqm=65.0,
    district="Алмалинский",
    floor=5,
    floors_total=12,
    condition="хороший"  # евроремонт / хороший / средний / требует ремонта / черновая
)
# → {fair_price_min: 45_000_000, fair_price_max: 53_000_000, verdict_guide: "..."}
```

**Файл:** [mcp_server/krisha_mcp.py](mcp_server/krisha_mcp.py)

## 4. RAG-пайплайн

RAG (Retrieval-Augmented Generation) используется для поиска похожих объявлений. Вместо того чтобы каждый раз скрапить Krisha.kz (медленно и ненадёжно), агент ищет похожие квартиры в локальной векторной базе.

### Компоненты и обоснование выбора:

**Chunking:** 1 объявление = 1 чанк. Объявление — атомарная единица поиска, делить его на части нет смысла. Текст чанка:
```
"2-комнатная квартира, Бостандыкский район, 65м², 
4/9 этаж, цена 42 000 000₸, цена м² 646 154₸"
```

**Embedding-модель: `text-embedding-3-small`**
- Стоимость: $0.02 за 1M токенов (в 5× дешевле `text-embedding-3-large`)
- Качество: достаточное для русскоязычных коротких описаний недвижимости
- Размер вектора: 1536 измерений

**Векторная БД: ChromaDB (локально)**
- Нет необходимости в регистрации и деплое (vs Pinecone, Qdrant Cloud)
- 36 686 эмбеддингов умещаются в памяти на ноутбуке
- Время поиска top-5: ~20мс

**Retrieval: top-5 по косинусному сходству**
- Запрос формируется автоматически: `"N-комнатная квартира <район> <площадь>м²"`
- 5 результатов достаточно для сравнения — больше перегружает контекст модели

### Как проиндексировать:
```bash
python3 rag/indexer.py
# Загружает все 37 763 объявления из SQLite батчами по 500
# Создаёт/обновляет коллекцию "krisha_listings" в chroma_db/
# Время: ~15 минут
```

### Как работает поиск:
```python
# rag/retriever.py
results = retrieve_similar_listings(
    query="2-комнатная квартира Бостандыкский 65м²",
    top_k=5
)
# → [{title, price, area, price_m2, district, url}, ...]
```

**Файлы:** [rag/indexer.py](rag/indexer.py), [rag/retriever.py](rag/retriever.py)

## 5. Скрапинг и данные

### База данных (SQLite)

37 763 объявления о продаже квартир в Алматы, собранные со страниц поиска Krisha.kz.

**Покрытие:**
- 8 районов: Алатауский, Алмалинский, Ауэзовский, Бостандыкский, Жетысуский, Медеуский, Наурызбайский, Турксибский
- Комнаты: 1, 2, 3, 4, 5 + без фильтра (студии, апартаменты)

**Схема:**
```sql
CREATE TABLE listings (
    listing_id   TEXT UNIQUE,  -- ID объявления на Krisha
    district     TEXT,          -- район Алматы
    rooms        INTEGER,       -- количество комнат (NULL = без фильтра)
    area         REAL,          -- площадь м²
    floor        INTEGER,       -- этаж
    floors_total INTEGER,       -- всего этажей в доме
    price        INTEGER,       -- цена в тенге
    price_m2     REAL,          -- цена м² (price / area)
    scraped_at   TEXT           -- дата сбора
)
```

**Как используется в агенте:**
При оценке цены функция `get_price_stats(district, rooms, area)` выбирает все объявления в том же районе с похожей площадью (±30%), удаляет выбросы методом IQR и возвращает медиану, среднее, P25 и P75. Это намного точнее, чем сравнивать со всеми квартирами района без учёта площади.

### Скрапер

```bash
python3 data/scraper.py
# или с ограничением страниц для теста:
python3 data/scraper.py --pages 5
```

**Особенности:**
- **Checkpoint-система:** прогресс сохраняется в `data/scraper_checkpoint.json` после каждой комбинации (район × комнаты). Если ноутбук выключился — запуск продолжится с того же места.
- **Дедупликация:** для прохода "без фильтра" загружает существующие ID и пропускает уже известные объявления.
- **Защита от блокировки:** случайная пауза 0.8–1.5 секунды между страницами.
- **Фильтрация выбросов:** объявления с ценой м² вне диапазона 200к–5млн₸ отбрасываются.

**Файлы:** [data/scraper.py](data/scraper.py), [data/database.py](data/database.py)

## 6. Мультимодальность — Vision анализ фото

**Зачем Vision, а не просто текстовое описание:**
Продавцы на Krisha.kz часто не описывают реальное состояние квартиры или преувеличивают его. Фото не лгут: состояние подъезда, трещины на стенах, советская сантехника — это то, что продавец не напишет в описании, но агент увидит на фото и учтёт в оценке цены.

**Как работает:**
1. Скачиваются все фото из объявления (до 10 штук) — и интерьер, и подъезд
2. Все фото отправляются в один запрос к GPT-4o Vision
3. Модель оценивает состояние по шкале 1–10 с жёсткими якорями:

| Оценка | Что означает |
|--------|-------------|
| 1–2 | Аварийное, жить невозможно без капремонта |
| 3–4 | Советский фонд, убитый подъезд, требует полного ремонта |
| 5–6 | Жилое, косметика, ничего не обновлялось 10+ лет |
| 7–8 | Свежий ремонт последних лет, современные материалы |
| 9–10 | Евроремонт или новостройка с дизайнерской отделкой |

4. **Важно:** корректировка цены рассчитывается детерминированно по score, не самой моделью — это устраняет галлюцинации в числах:

| Score | Корректировка к цене |
|-------|---------------------|
| 9–10 | +10% |
| 7–8 | +3% |
| 5–6 | 0% |
| 3–4 | −8% |
| 1–2 | −15% |

5. Если фото слишком тёмные или размытые — флаг `low_confidence: true`, корректировка = 0% (не штрафуем за плохие фото).

**Файл:** [agent/nodes.py](agent/nodes.py) — функция `analyze_photos`

## 7. Мониторинг — LangSmith

**Зачем LangSmith:**
Без мониторинга нельзя понять почему агент ошибся — какая нода вернула неверные данные, сколько токенов потратила Vision, где была задержка. LangSmith автоматически записывает каждый LLM-вызов и каждую ноду LangGraph как отдельный span с inputs/outputs, latency и стоимостью.

**Настройка (в [config.py](config.py)):**
```python
os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_PROJECT"] = "krisha-agent"
os.environ["LANGCHAIN_API_KEY"] = LANGSMITH_API_KEY
```

После этой строчки **все** вызовы LangGraph, LangChain и OpenAI через langchain автоматически логируются — никаких дополнительных изменений в коде не нужно.

**Что видно в дашборде:**
- Полное дерево нод каждого запроса: `LangGraph → fetch_listing → search_similar → ...`
- Inputs и outputs каждой ноды
- Время выполнения каждого шага
- Количество токенов и стоимость каждого LLM-вызова
- История всех запросов

**Статистика:** 100 трейсов, 12 полных запусков агента в `krisha-agent` проекте.

Открыть дашборд: [smith.langchain.com](https://smith.langchain.com) → Projects → krisha-agent

## 8. Эвалюации и A/B тест

### Golden Dataset

30 тестовых примеров в [evals/golden_dataset.json](evals/golden_dataset.json). Каждый пример содержит:
- `listing_data` — данные квартиры (район, площадь, цена, этаж, год)
- `price_stats` — рыночная статистика по этому типу квартир
- `photo_analysis` — предрассчитанный результат Vision анализа
- `expected.verdict` — ожидаемый вердикт (выгодно / дорого / рыночная цена)
- `expected.key_factors` — ключевые факторы, которые должны быть в рекомендации

**Покрытие edge cases:**
- Квартиры на первом и последнем этажах (дисконт -5% и -3%)
- Дома до 1950 года (дисконт -35%)
- Новостройки 2023–2024 года (премия +8%)
- Панельные vs монолитные дома
- Аварийное состояние (оценка 1–2, дисконт -15%)
- Евроремонт в старом доме

### Метрики

**Verdict Accuracy** — доля правильно определённых вердиктов:
```
accuracy = правильных_вердиктов / всего_примеров
```
Вердикт вычисляется детерминированно алгоритмом, не LLM — поэтому точность стабильна.

**Relevance Score (0–1)** — покрывает ли рекомендация ключевые факторы:
```
relevance = упомянутых_факторов / всего_факторов_в_expected
```
Проверка через keyword matching по ключевым словам и числам из каждого фактора.

**LLM-as-Judge (1–5)** — GPT-4o-mini независимо оценивает качество рекомендации: конкретность, наличие цифр, корректность вердикта.

### Запуск

```bash
# Config B (production — gpt-4o для рекомендаций)
python3 evals/run_evals.py \
    --dataset evals/golden_dataset.json \
    --output evals/results_config_b.json \
    --config B

# Config A (baseline — gpt-4o-mini для рекомендаций)
python3 evals/run_evals.py \
    --dataset evals/golden_dataset.json \
    --output evals/results_config_a.json \
    --config A
```

### Результаты A/B теста

| Config | Модель рекомендации | Verdict Accuracy | Relevance | LLM Judge |
|--------|--------------------|-----------------:|----------:|----------:|
| A (baseline) | gpt-4o-mini | 86.7% | 0.851 | 4.97 / 5 |
| B (production) | gpt-4o | 86.7% | 0.794 | 4.90 / 5 |

**Вывод:** Accuracy одинакова (вердикт детерминирован), gpt-4o-mini незначительно лучше по relevance. Для рекомендаций достаточно gpt-4o-mini — это снижает стоимость запроса в 5–10×. GPT-4o оставлен только там, где он действительно нужен — Vision анализ фото.

## 9. Выбор моделей и гиперпараметры

### Модели

| Задача | Модель | Обоснование |
|--------|--------|-------------|
| Анализ фото (Vision) | `gpt-4o` | Единственная модель OpenAI с Vision API нужного качества |
| Текстовая рекомендация | `gpt-4o` | Подтверждено A/B тестом: качество не хуже mini |
| Follow-up вопросы | `gpt-4o-mini` | Простые Q&A в контексте, не требуют топ-модели |
| Эмбеддинги | `text-embedding-3-small` | Лучший баланс цены и качества для коротких русских текстов |

**Почему не Claude / Gemini:**
- Claude не доступен через OpenAI SDK, требует отдельной интеграции
- Gemini Vision показал худшие результаты на русскоязычных подписях к фото в тестах
- GPT-4o — проверенный стандарт для Vision задач с хорошей русской локализацией

**Стоимость одного запроса:**
- Vision (10 фото): ~$0.08–0.12
- Рекомендация: ~$0.01–0.02
- Эмбеддинги: ~$0.0001
- **Итого: ~$0.10–0.15 за запрос**

### Гиперпараметры

| Параметр | Нода | Значение | Обоснование |
|----------|------|---------|-------------|
| `temperature` | analyze_photos | 0.1 | Нужна строгость, не творчество — оценка состояния квартиры |
| `temperature` | generate_recommendation | 0.4 | Нужен связный нарратив, небольшая вариативность допустима |
| `temperature` | human_review | 0.3 | Баланс точности и естественности в диалоге |
| `top_p` | все | 0.9 | Стандарт, убирает хвост маловероятных токенов |
| `max_tokens` | analyze_photos | 600 | Достаточно для JSON с оценкой и списком проблем |
| `max_tokens` | generate_recommendation | 800 | 5 разделов рекомендации умещаются в ~600 токенов |

**Файл:** [config.py](config.py)

```bash
python3 -m venv venv
source venv/bin/activate      # macOS / Linux
# venv\Scripts\activate.bat   # Windows
```

### Шаг 3 — Установить зависимости

```bash
pip install -r requirements.txt
```

### Шаг 4 — Настроить переменные окружения

```bash
cp .env.example .env
```

Открыть `.env` и заполнить:

```env
OPENAI_API_KEY=sk-...           # обязательно
LANGSMITH_API_KEY=lsv2_...      # обязательно для мониторинга
LANGSMITH_PROJECT=krisha-agent  # имя проекта в LangSmith
```

### Шаг 5 — Запустить

```bash
streamlit run app.py
```

Открыть: `http://localhost:8501`

> **База данных SQLite уже включена в репозиторий** (`data/krisha_listings.db`, 37k объявлений).  
> ChromaDB нужно создать один раз командой ниже (занимает ~15 минут).

### Опционально: обновить базу данных

```bash
# Пересобрать базу с нуля (занимает несколько часов)
python3 data/scraper.py

# Переиндексировать RAG после обновления базы
python3 rag/indexer.py
```

### Опционально: запустить MCP-сервер отдельно

```bash
fastmcp run mcp_server/krisha_mcp.py
```

## Структура проекта

```
krisha-agent/
│
├── app.py                        # Streamlit UI — точка входа
├── config.py                     # Модели, температуры, API ключи, RAG параметры
│
├── agent/
│   ├── graph.py                  # LangGraph граф: ноды, рёбра, ветвления
│   ├── nodes.py                  # Логика каждой из 6 нод
│   └── state.py                  # AgentState TypedDict
│
├── mcp_server/
│   └── krisha_mcp.py             # FastMCP сервер: 3 tool'а
│
├── skills/
│   └── krisha_analyzer/
│       └── SKILL.md              # Описание агента: триггеры, входы, выходы
│
├── rag/
│   ├── indexer.py                # Индексация SQLite → ChromaDB
│   └── retriever.py              # Поиск top-5 по косинусному сходству
│
├── data/
│   ├── database.py               # SQLite: get_price_stats, upsert_listing

│   └── results_config_b.json     # Результаты Config B (gpt-4o)

├── ARCHITECTURE.md               # Подробная архитектурная схема
```
