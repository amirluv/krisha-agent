# KrishaAgent — Результаты Эвалюаций

## Методология

### Golden Dataset
- **Размер:** 30 примеров
- **Покрытие:** 8 районов Алматы, 1–4 комнатные квартиры
- **Распределение вердиктов:** 11 × выгодно, 13 × дорого, 6 × рыночная цена
- **Каждый пример содержит:** `listing_data`, `price_stats`, `photo_analysis`, ожидаемый вердикт + ключевые факторы

### Метрики

#### 1. Verdict Accuracy
Процент правильно определённых вердиктов (выгодно / дорого / рыночная цена).  
Вердикт — детерминированный результат ценовой логики, не зависит от модели.

#### 2. Relevance Score (0–1)
Доля ключевых факторов из `expected.key_factors`, которые упомянуты в тексте рекомендации.  
Метод: keyword matching по ключевым словам и числам из каждого фактора.

#### 3. LLM Judge Score (1–5)
GPT-4o-mini оценивает качество рекомендации независимо:
- **5** — вердикт верный, объяснение конкретное с цифрами, все факторы учтены
- **4** — вердикт верный, хорошее объяснение, не все факторы
- **3** — вердикт верный, но объяснение поверхностное
- **2** — вердикт неверный, объяснение логичное
- **1** — вердикт неверный и объяснение неудовлетворительное

## Выводы A/B теста

### Что показал эксперимент

**Вердикт не зависит от модели.** Accuracy одинакова (86.7%) в обоих конфигах — это закономерно: вердикт вычисляется детерминированно через алгоритм сравнения цены с рынком, LLM его не формирует.

**gpt-4o-mini незначительно лучше** по relevance (+0.057) и LLM judge (+0.07). Разница в пределах статистической погрешности при выборке 30, но тенденция устойчивая.

**Практический вывод:** для продакшена оправдано оставить `gpt-4o` для Vision-анализа фото (задача требует мультимодального понимания) и перейти на `gpt-4o-mini` для текстовых рекомендаций — это снизит стоимость запроса в ~5-10 раз без потери качества.

### Ошибочные вердикты (4/30 в каждом конфиге)

Все 4 ошибки — одинаковые в обоих конфигах, что подтверждает: ошибки в ценовом алгоритме, не в тексте рекомендации.

| ID | Ожидалось | Получено | Причина |
|----|-----------|----------|---------|
| eval_003 | рыночная цена | дорого | Граничный случай: панельный дом с -10%, IQR сдвигает диапазон |
| eval_010 | рыночная цена | дорого | Аналогично: панель 1980 года получила повышенный дисконт |
| eval_019 | рыночная цена | выгодно | Первый этаж + монолит 2017 — суммарная корректировка смещает в выгодно |
| eval_020 | рыночная цена | дорого | Панельный дисконт -10% при цене чуть выше медианы |

**Наблюдение:** граничные "рыночная цена" случаи наиболее уязвимы — они находятся в зоне ±8% от справедливой цены, где небольшие корректировки меняют категорию.

## Анализ метрик

### Relevance Score: почему 0.79–0.85, а не 1.0

Keyword matching — упрощённый метод: агент может упомянуть фактор другими словами. Например:
- Ожидалось: `"1985 год -5%"` → ищем `"1985"` и `"год"` в тексте
- Агент пишет: `"дом 85-го года"` → число "1985" не найдено, хотя смысл передан

Реальное покрытие факторов выше числового показателя.

### LLM Judge: почему почти всегда 5/5

GPT-4o-mini оценивает структурированность и конкретность — агент всегда даёт цифры (цена м², разница с рынком, корректировки в %). Это ценится выше, чем стиль изложения.

## Запуск эвалюаций

```bash
# Config B (production, gpt-4o)
python3 evals/run_evals.py \
    --dataset evals/golden_dataset.json \
    --output evals/results_config_b.json \
    --config B

# Config A (baseline, gpt-4o-mini)
python3 evals/run_evals.py \
    --dataset evals/golden_dataset.json \
    --output evals/results_config_a.json \
    --config A
```

Результаты сохранены в: `evals/results_config_a.json`, `evals/results_config_b.json`
